# Objective

Define the NormalizedHap interface and normalizeStrudelHap() function, then update PatternScheduler.query() to return NormalizedHap[] and wire StrudelEngine's scheduler wrappers to normalize raw Strudel haps at the query boundary.

# Context

Sketches currently reach into raw Strudel hap internals (`hap.whole.begin`, `hap.value.note`, `hap.value.color`). This couples every viz consumer to Strudel's internal shape, making non-Strudel engines impossible to plug in. The normalization point is PatternScheduler.query() -- it's the single boundary between engines and viz consumers.

Key invariants:
- PV1: Strudel Pattern methods return new instances (normalization reads from queryArc output, never mutates)
- UV6: Observation without mutation (normalize is a read-only map)
- New PV6 (proposed): NormalizedHap is a value object -- plain, no methods, no prototype chain

Key lifecycle (PK1): StrudelEngine.evaluate() captures patterns, builds trackSchedulers from them. The `.map(normalizeStrudelHap)` wraps at the query() call site inside those schedulers.

       },
       ```

       query: (begin: number, end: number) => {
         try { return captured.queryArc(begin, end).map(normalizeStrudelHap) } catch { return [] }
       },
       ```

    These are the ONLY two places raw Strudel haps cross the boundary to viz consumers.
    After this change, all PatternScheduler.query() calls return NormalizedHap[].
  </action>
  <ownership>
    Raw Strudel haps: created by pattern.queryArc() -> transformed by normalizeStrudelHap() -> consumed as NormalizedHap by sketches

  </ownership>

  </lifecycle>
